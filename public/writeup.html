SE Content Hub: what it took to give the library a brain

One upload pipeline, one AI assistant, one LaunchDarkly control plane. Every deck, demo, video, and doc your team has ever shared — instantly queryable in plain English, access-controlled by a flag edit, no redeploy required. Notes on the architecture, what surprised us, and what we'd do differently.

Solutions Engineering campaign  ·  internal

What we tried to prove

That a solutions engineering team can build its own intelligent content hub without a dedicated data team — and without redeploying every time library access rules change. Concretely: an SE uploads a sales asset (deck, demo, video, doc, code snippet) and within a few seconds it's extracted, chunked, and embedded; an AI chat assistant reads from those embeddings to answer questions like "what's our best competitive deck for a security-focused fintech prospect?"; a LaunchDarkly feature flag controls which items are visible per user — and because the chat assistant draws from the same filtered pool, one flag edit scopes both the gallery and the AI simultaneously, with no code deploys.

Secondary claim: AI-generated metadata is good enough to replace manual tagging. On upload, GPT-4o reads the extracted text and writes title, description, and tags back to the database. SEs spend zero time on taxonomy.

Three surfaces, one pipeline, one control plane. The insight that made the system debuggable: the LaunchDarkly flag doesn't just filter the UI — it filters the embedding lookup too. One source of truth for access, not two.

Three surfaces, three clean ownership lines. The Upload Pipeline owns ingestion — it writes to Supabase and calls OpenAI. The Chat Assistant owns retrieval — it reads from pgvector and calls GPT-4o. The Library Gallery owns browse and access. The LaunchDarkly flag sits above all three: when a flag rule changes, both the gallery and the AI's retrieval pool update at once, because they query the same filtered set. No surface can accidentally show content another surface has already hidden.

Results

Full upload-to-searchable pipeline in under 30 seconds for most file types. AI chat correctly surfaces relevant assets by meaning rather than keyword on the first response in the majority of test queries. Flag-controlled content segmentation proved out as a zero-deploy operation — adding a new content pool or restricting access to a region is a LaunchDarkly UI edit.

The thing that took longest wasn't AI

It was text extraction. PDFs are fine. PowerPoint files are fine. But PPTX files that were originally exported from Keynote, DOCX files with heavy image embeds, videos that are screen recordings with no audio — each one has its own failure mode. We ended up with six distinct extraction paths before the ingestion pipeline was reliably solid:

pdfjs-dist for PDFs  ·  JSZip XML parsing for PPTX/DOCX  ·  GPT-4o Vision for image files  ·  OpenAI Whisper for audio/video  ·  file.text() for plain text/code  ·  metadata fallback for everything else

The metadata fallback matters more than it sounds. When extraction fails or the file is binary-only, GPT-4o can still generate reasonable title and tags from just the filename, file type, and any description the uploader provided. Content that would otherwise be invisible to search stays partially discoverable.

The other surprise: the AI metadata generation is genuinely better than manual tagging. GPT-4o reads the full extracted text, not just the filename, and writes tags that reflect actual content — competitor names mentioned in the deck, use cases covered, industries referenced. Manual tagging would never be that thorough.

Patterns worth carrying out of this

The flag has to control the retrieval, not just the display

The naive implementation gates the gallery UI on a flag and leaves the AI chat searching everything. That's a correctness bug. A user who can't see a restricted deck in the gallery can ask the chat assistant about it and get an accurate answer. We connected the two from the start: the chat's match_content RPC receives the same filter_user_id and content-set parameters that the gallery uses. One flag, both surfaces, same answer.

Chunk strategy is the single biggest lever on answer quality

We landed on 2,000-character chunks with 200-character overlap. Too small and a single slide's context doesn't fit; too large and unrelated sections get ranked together, diluting relevance. The deduplication step — keeping only the best-scoring chunk per source before passing context to GPT-4o — was the other key improvement. Without it, a long deck would fill the context window with redundant chunks from the same file.

Persistent chat history makes the assistant dramatically more useful

A stateless chatbot resets on every message. Storing sessions in Supabase means an SE can ask a follow-up three days later and the assistant remembers the conversation. The implementation is a single chat_sessions table with one row per user and a chat_messages join — small schema surface, large usability delta.

Optimistic UI removes the most painful latency

Embedding a file takes 5–15 seconds. Showing the item in the gallery immediately (before the embedding completes) and surfacing an embedding_status indicator on the card means the upload feels instant even when the AI pipeline is still running. Users don't wait at a spinner; they see their content appear and the AI catch up in the background.

Where this can go from here

Three reasonable next destinations, ordered by lift.

  1. AI Configs for prompt and model control

    The chat assistant's system prompt and model selection are currently hardcoded. Wrapping them in a LaunchDarkly AI Config would let the team A/B test prompt variants, swap models between GPT-4o and other providers without a redeploy, and run the Agent Optimization SDK to iterate on response quality automatically — all from the LD UI. This is the same control-plane pattern proven in the Moonshot; the plumbing already exists.

  2. Semantic deduplication at upload time

    If two SEs upload the same deck independently, both appear in the gallery and both compete for the same embedding-search slots. An embedding similarity check at upload time — flag a new item when cosine distance to an existing item exceeds a threshold — would surface duplicates before they're committed, keeping the library clean without manual curation.

  3. Automated content freshness scoring

    A nightly job could re-score every item against a "recency and relevance" rubric — embedding distance to recent uploads, view-count trajectory, age — and surface a "may be outdated" signal on cards. Combined with an LD flag that hides items below a freshness threshold, the library self-prunes without anyone manually archiving old decks.

Smaller branches worth a sentence: per-region content pools (the flag already supports attribute-based rules, so a region user attribute is all that's missing); an Ideas-to-content promotion flow that notifies the uploader when an AI-generated draft was turned into a published item; multi-modal search that queries against video transcript embeddings alongside slide text.

Bottom line

The project is demo-complete. Upload pipeline works across all major file types. Semantic search returns meaningful results. AI chat is context-aware and persistent. LaunchDarkly controls both the gallery and the AI in one flag edit. The headline claim — that a team can build an intelligent internal content platform with LaunchDarkly as the access control spine, no custom ACL logic required — is supported by a working system.

The next interesting move is AI Configs. The chat assistant is already good; wrapping it in an AI Config turns "good" into "continuously improvable from the UI."
